# 9. 검증 및 측정
- 에이전트 기반 시스템을 평가하는 방법론 살펴보기
  - 주요 원칙, 측정 기법, 검증 전략을 다룸
  - 단순한 기능을 넘어 정확성, 일관성, 개연성, 반응성을 포함한 에이전트 출력 신뢰성은 정밀 조사가 필요
### 9.1 에이전틱 시스템의 측정
- 명확한 목표를 정의 및 유의미한 지표를 수립하여 체계적인 평가 프로세스 적용 필요
  - 높은 성능과 사용자 만족 달성
##### 9.1.1 측정: 에이전틱 시스템의 핵심
- 효과적인 측정은 목표와 요구사항에 부합하는 명확하고 실행 가능한 지표 식별
  - 해당 지표는 에이전트 작업 수행 능력과 사용자 기대 충족여부를 평가하는 멘치마크 역할
- 성공 여부는 시스템이 바라는 결과를 반영하는 목표를 정의하는데 달림
  - 우선 순위가 높은 대표적인 사용 사례를 구성
- 올바른 지표 선택도 중요
  - 정확성, 응답 시간, 견고성, 확장성, 정밀도, 재현율 같은 정량적 지표
  - 사용자 만족도 같은 정성적 척도
- 언어 기반 에이전트 경우 기존의 완전 일치 지표는 무쓸모
  - 임베딩 기반 거리, BERTScore, BLEU 등 의미론적 유사도 측정이 더 좋음
- 측정의 이점을 실현하려면 평가 메커니즘을 에이전트 개발 라이프사이클에 직접 통합해야함
  - 지속적인 피드백과 변화하는 요구사항에 맞춰 에이전트와 지표를 모두 개선
##### 9.1.2 개발 라이프사이클에 평가 통합
- 선도된 팀은 개발의 모든 단계에 자동화된 오프라인 평가를 통합
  - 에이전트에 새로운 도구나 워크플로가 추가될 때마다 그에 상응하는 `테스트 케이스`와 `평가 예제`를 평가 세트에 추가해야함
- 고품질 평가 세트는 `에이전트가 처리해야할 작업 명세서 역할` 및 `재현성과 회귀 감지 지원`
##### 9.1.3 평가 세트 생성 및 확장
- 모든 측정 전략의 토대는 고품질 평가 세트
  - 직면할 다양성, 모호성, 엣지 케이스를 반영
  - `정적이고 수작업으로 선별된 테스트 스위트는 최신 에이전틱 시스템에 불충분`
- 좋은 평가 세트는 입력 상태와 예상 결과를 모두 정의 -> 에이전트 행동을 자동으로 검증
- 아래와 같은 목표 지향적 생성 기법 적용 가능 -> 미묘한 오류 발견 감지 가능
  - 적재적 프롬프팅(에이전트가 모순된 말을 하게 만드는 사용자 메시지 찾기)
  - 반사실적 편집(프롬프트에서 단어 하나 바꾸고 에이전트가 실패하는지 확인)
  - 분포 보간(두 가지 의도를 섞어 의도적으로 모호한 요청 만들기)
### 9.2 컴포넌트 평가
- 잘 설계된 단위 테스트는 의도한 대로 작동하는지 확인해 에이전트의 전반적인 신뢰성과 성능에 기여
##### 9.2.1 도구 평가
- 도구에 대한 고품질 단위 테스트는 가능한 사용 사례를 빠짐없이 나열하는 것에서 시작
  - 위험한 엣지 케이스나 좋지 못한 시나리오까지 포함
- 성숙한 에이전트 개발 프로세스는 각 도구마다 자동화된 테스트 모음을 정의 -> 도구가 부하나 장애 상황에서도 우아하게 성능이 저하되도록 보장
  - 출력의 정확성
  - 지연 시간
  - 리소스 사용량
  - 오류처리
- 도구 테스트는 같은 입력에 대해 출력이 결정적이어야 한다는 점을 검증
  - 도구 설계상 확률성이 포함되어 있으면 결정적일 필요 없으며, 통계적 특성 점검 필요
  - API 등 외부 의존성이 있는 도구의 경우, 잘못 처리되거나 엣지 케이스 재현 필요 -> 모킹이나 시뮬레이터 사용
  - 회귀 테스트 중요 -> 도구 수정 시, 과거 기능 망가지지 않는지 확인 필요
##### 9.2.2 계획 능력 평가
- 에이전트형 계획은 종종 확률적이거나 적응적이어서 취약하거나 일관성 없는 작동을 피하기 위해 더욱 세심한 테스트가 필요함
  - 실행 중에 학습한 내용에 따라 도구 호출을 순서대로 구성하고, 필요하면 중간에 실행을 중단해야할 수 도 있음
- 계획의 품질을 평가하기 위해 정형화된 워크플로부터 시작
  - 잘 알려진 일반적인 사용자 의도와 그에 대해 검증된 에이전트의 응답 쌍으로 구성
  - 각 시나리오마다 초기 환경, 대화 이력, 도구 사용 등 기대되는 결과를 인코딩
  - 해당 플랜을 체계적으로 평가하기 위해 에이전트를 엔드투엔드로 실행하고 시나리오를 정랍 레이블과 비교
    - 도구 재현율: 계획 모듈이 예상한 도구를 모두 호출했는가?
      - 해당 값이 낮으면 필수적인 행동 수행하지 못함을 의미
    - 도구 정밀도: 불필요한 도구 호출을 피했는가?
      - 해당 값이 낮으면 목표를 잘못 이해했거나 사용자의 의도를 잘못 해석했음을 시사
    - 파라미터 정확도: 각 도구에 주문 ID나 환불 금액처럼 올바른 인자를 제공했는가?
      - 잘못된 컨텍스트에 기반해 작동했다는 사실 시사
- 계획 과정에서는 종종 컨텍스트에 의존하기 때문에 엣지 케이스를 테스트하는 것이 특히 중요
  - 사용자가 모호한 입력을 제공하거나 여러 메시지에 걸쳐 스스로 모순되는 발언을 한다면 어떨까?
- 계획 모듈은 일관성에 대해서도 평가해야함
  - 결정적 시나리오에서는 동일한 입력이 항상 동일한 출력을 만들어야함
- `계획 능력 평가는 에이전트가 무엇을 해야 하는지 제대로 알고 있는지를 알려줌`
  - 사용자 의도 이해 및 컨텍스트에 잘 맞는 행동으로 변환할 수 있는지 확인
##### 9.2.3 메모리 평가
- 메모리 테스트는 단순한 저장 조회 정확성 검증뿐만 아니라 메모리가 커짐에 따라 무결성, 관련성, 효율성까지 보장 필요
- 테스트
  - 정확성
    - 메모리에 기록된 데이터가 정확히 저장되었는지 확인
    - 시간이 상당히 지난 뒤나 다른 작업이 개입한 뒤에도 정밀하게 다시 조회 가능한지 확인
    - 시스템 견고성 확인을 위해 잘못된 형식, 중복, 모호한 항목 등으로 스트레스 주는 테스트 수행 필요
  - 관련성
    - 조회 로직이 오래되었거나 무관한 정보를 표면으로 끌어올리지 않도록 해야함
  - 효율성
    - 메모리 부하가 증가하는 상황에서 조회 시간과 리소스 사용량을 벤치마킹 해 병목을 찾아야함
  - 복원력
    - 이슈 영향력 최소화 -> 복구 또는 통제된 방식으로 실패 처리
##### 9.2.4 학습 평가
- 확률적 특성과 데이터 의존성 때문에 단위 테스트가 복잡한 편 -> 그래도 엄격하게 해야함
  - 시간이 지나면서 이미 익힌 행동을 '잊어버리는' 일이 없도록 보장해야함
- 테스트
  - 기본 학습 루프 검증
    - 레이블 데이터, 피드백 등에 따라 에이전트가 자신의 파라미터, 캐시, 규칙을 올바르게 갱신했는가 확인
      - 지도학습: 데이터셋을 학습했을 때 기대한 정확도에 도달했는지 테스트
      - 강화학습: 시간이 지남에 따라 실제로 행동 개선으로 이어지는지, 정체 구간을 감지해 적절히 처리하는지 확인
  - 일반화
    - 학습한 행동을 학습데이터의 분포와 다른 분포를 가지는 시나리오에 얼마나 잘 적용하는지 평가
  - 적응력
    - 사용자 입력, 도구 실패 등에서 에이전트가 치명적인 망각이나 성능 붕괴 없이 적응할 수 있는지 확인 필요

### 9.3 총체적 평가
- 통합 테스트는 에이전트 기반 시스템의 전체를 평가하여 모든 하위 시스템에 매끄럽게 함께 작동하는지 확인
  - 단위 테스트는 개별 컴포넌트를 고립된 상태에서 정확성 검증
  - 한 모듈의 출력이 다른 모듈의 입력이 되는 일이 잦은 에이전트 기반 시스템에서는 필수
##### 9.3.1 엔드투엔드 시나리오에서의 성능
- 실제 사용에 가까운 조건에서 시스템이 시작부터 끝까지 완전한 작업을 수행하는 능력을 검증
  - 올바른 행동을 선택하는지뿐 아니라 명확하게 소통하고 사용자 의도와의 정합성을 유지하는지도 확인
- 에이전트는 구조화된 입력(데이터 및 대화 이력 포함)을 받아 출력이 기대 최종 상태와 일치하는지 비교
  - 어떤 도구, 어떤 파라미터, 최종 메시지 필요한 문구 검증
  - 결과는 도구 재현율, 도구 정밀도, 파라미터 정확도, 문구 재현율, 통합 작업 성공 점수 등 지표로 요약
- 자동화된 테스트의 품질이 그에 사용되는 평가 세트와 지표의 품질에 좌우됨
  - 테스트 케이스가 지나치게 좁거나 대표성이 부족하면? 오프라인 테스트에는 성능이 좋지만 실제는 실패
- `평가를 고정된 체크리스트가 아닌 살아 있는 프로세스로 다루는 것이 최선`
  - 테스트 세트를 정기적으로 확장하고 개선 필요
  - 평가 방법과 지표를 반복적으로 다듬어 에이전트 측정 필요
##### 9.3.2 일관성
- 확률적이고 비결정적인 모델에 의존을 하기에 일관성 테스트는 특히 어려움
  - 출력이 입력과 정합성을 유지
  - 장기간의 교환에서도 일관된 흐름을 이어 가기
  - 사용자가 의도한 질문이나 작업을 안정적으로 처리하는지 확인 중점
    - 예) 특정 도구 호출 전 파손 사진을 요청 -> 상호 작용이 길어져도 모순되지 않게 진행
- 핵심 목표 중 하나는 다양한 시나리오에서 에이전트 응답이 주어진 입력과 정합성을 유지하는지 검증
  - 긴 상호작용은 추가 복잡성을 야기 -> 성능 저하(이전 발언과 모순되거나 주제에서 벗어난 응답)
  - 시스템이 시간 경과에도 일관된 성능을 유지할 수 있는지 평가하기 위해 요구
- 자동 평가가 드물지만 치명적인 엣지 케이스를 놓칠 수 있음
  - 지속적인 수동 점검과 평가 데이터의 주기적 갱신이 중요
- `자동화와 수동 리뷰는 모두 필수적`
  - 사람 리뷰는 미묘한 불일치 평가와 응답이 의도된 목적에 얼마나 부합하는지 피드백 제공
- 액터-크리틱 접근법도 일관성 테스트에 유용
##### 9.3.3 응집성
- 에이전트 출력이 상호작용 전반에 걸쳐 논리적이고 컨텍스트에 부합하며 일관되게 유지되는지 확인
  - 사용자가 같은 말을 반복하지 않아도 이전 정보를 참조
  - 예) '깨진 머그컵' 시나리오에서 응집성은 환불할 때 초기 파손 신고와 사진 업로드 참조하고 다품목 주문 세부 정보를 놓치지 않는 것
- 테스트는 장기 상호 작용을 시뮬레이션해 에이전트가 상태에 대한 일관된 이해를 유지 하는지 행동이 논리적이며, 목표 지향적 순서를 따르는지 검증
- 응집성 테스트는 실환경에서 에이전트 기반 시스템의 신뢰, 사용성, 실질적 가치를 유지하는 데 필수
##### 9.3.4 할루시네이션
- 에이전트가 틀렸거나 의미가 없거나 지어낸 정보를 생성할 때 발생
  - 응답이 일관되게 생성되도록 엄격한 테스트와 완환 전략이 필요
    - `신뢰할 수 있는 데이터에 근거해 출력을 고정`
- 핵심은 콘텐츠 정확성을 보장
  - 허구가 아니라 사실 데이터를 기반하는지 검증 -> 검증된 데이터에 의존 처리
- 데이터 의존성은 할루시네이션 해결의 핵심 요소
  - 데이터 소스의 품질의 좌우 -> 항상 정확하고 관련성 있으며 최신의 소스에서 정보를 가져오도록 보장
- 피드백 루프는 할루시네이션 탐지와 해결에 필수
  - 부정확성을 표시하고 검토 및 교정 -> 데이터 소스 업데이트 트리거
- 할루시네이션 대응은 하이브리드 방식(사람-AI 피드백 루프) 중심으로 발전
  - AI와 협업하여 사용자 인지 부하를 줄이며, 실시간 감독을 통해 허구 정보를 퍼지기 전에 교정
- 콘텐츠 정확성에 우선순위를 두고 데이터 의존성을 엄격히 관리, 피드백 메커니즘을 활용, 다양한 시나리오 철저히 테스트 -> 위험 최소화 가능
##### 9.3.5 예기치 않은 입력
- 학습이나 설계 가정 박에 있는 입력을 의도적으로 제공하여 테스트
  - 에이전트가 중단되거나 유해한 출력을 만들지 않고 상황에 맞게 정중히 대처
- 효과적인 테스트로 무작위 '퍼징'뿐만 아니라 과거 사고나 적대적 분석에서 얻은 통찰로 엣지케이스를 체계적으로 탐색
### 9.4 배포 준비
- 운영 환경에서의 신뢰성과 신뢰도를 보장하기 위해 엄격한 준비성 점검과 품질 게이트 필요
  - 테스트 통과를 넘어 시스템이 의도한 기능을 안전하고 일관되며 효율적으로 수행하는지도 평가
- 명확한 배포 기준을 수립하는 것이 첫걸음
  - 실무에서 구조화된 체크리스트를 사용해 모든 컴포넌트가 엄격히 테스트되고 리뷰되었는지 확인 필요
- 강제하는 핵심 장치는 게이트 메커니즘
  - 모든 요구사항이 충족되기 전에는 프로덕션 승격을 막는 자동 또는 수동 점검
- 새버전을 신뢰성 있게 롤 아웃하고 출시 후 회귀를 모니터링하며, 이슈 발생에 롤백하는 프로세스 수립 필요
