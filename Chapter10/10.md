# 10 운영 환경 모니터링
- 모니터링을 잘하면 더 빨리 학습하고 더 안전하게 배포하며 신뢰성을 개선함
### 10.1 모니터링: 학습의 출발점
- 에이전트 실패는 미묘하기에 모니터링은 필수
  - LLM은 유창하지만 사람의 오도함
  - 계획은 부분적으로 작동해 목표를 놓침 -> 불일치가 일어나지만 시스템 오류는 발생하지 않음
- `에이전트 실패를 하나의 장애로 생각하지 말고 테스트 케이스로 봐야함`
  - 운영에서 실패할 때마다 해당 시나리오는 회귀 테스트로 전화되어야함
  - 복잡한 사례를 잘 처리한 케이스도 골든 패스로 삼기
  - 실패와 성공 케이스로 CI/CD 코퍼스를 갖추게 됨
- 에이전트같은 확률적 시스템은 모니터링의 핵심은 진짜 실패와 기대 가능한 변동을 구분하는 일
  - 진짜 실패: 수정이 필요한 체계적 문제
  - 기대 가능한 변동: 출력이 다르지만 허용 가능한 비결정성
- 효과적 모니터링은 인프라 신호와 의미적 행동을 모두 포괄
  - 의미적 행동: 의도 파악, 도구 선택, 할루시네이션, 작업 중도 포기
- 계층적 피드백 루프 구축 필요
  - 런타임 이벤트(도구 호출, 생성, 폴백)를 계측해 로그, 트레이스, 시각화 같은 백엔드로 스트리밍
### 10.2 모니터링 스택
- 어떤 모니터링을 선택하느냐에 에이전트 시스템의 개발 속도가 달림
  - 기존 인프라 지표뿐 아니라 할루시네이션률, 도구 효과, 사용자 입력의 분포 등 의미적 신호도 포착 필요
##### 10.2.1 그라파나
- 기존 구성 가능성이 높아 에이전트를 중심으로 맞춤형 가시성을 구축하려는 팀에 유연한 선택지
  - 설정과 통합
    - 랭그래프 애플리케이션에서 오픈텔레메트리를 초기화해 스팬(도구 호출,LLM 생성)과 지표(토큰 사용량)을 내보냄
  - 핵심 기능
    - 의미적 지표의 실시간 대시보드, 재시도 급증 같은 이상 징후 알림 등을 제공
  - 트레이드 오프
    - 장점
      - 유연성과 벤더 종속성이 없음
    - 단점
      - 멀티툴 환경(로키/템포 별도 운영)과 인프라가 아닌팀에게 더 가파른 학습 곡선
##### 10.2.2 ELK 스택
- 기존 엔터프라이즈 환경을 AI 워크로드로 확장하는데 자주 쓰임
  - 설정과 통합
    - 노드를 계측해 구조화 이벤트를 로그로 남기고 엘라스틱 서치의 머신러닝 작업으로 에이전트 출력의 이상 징후를 탐지
  - 핵심 기능
    - LLM 출력에 대한 고급 전문 검색과 벡터 검색, 예측 알림 지원
  - 트레이드 오프
    - 장점
      - 우수한 검색, 분석 기능과 엔터프라이즈급 확장성
    - 단점
      - 높은 리소스 요구(엘라스틱서치는 메모리 집약적)
      - 더 복잡한 배포(다수 서비스 운영)

##### 10.2.3 어라이즈 피닉스
- LLM 트레이싱과 평가에 초점을 맞춘, 에이전트 모니터링을 확장하는 디버그 지향 도구
  - 설정과 통합
    - SDK로 랭그래프를 계측
    - 하이브리드 활용을 위한 오픈텔레메트리로 내보내기 지원
  - 핵심 기능
    - 평가를 포함한 구조화 트레이싱
  - 트레이드 오프
    - 장점
      - 평가/디버깅 특화와 프로토타입에 적합한 경량화
    - 단점
      - 트레이스/평가에 제한
##### 10.2.4 시그노즈
- 지표, 트레이스, 로그를 하나의 도구로 통합한 오픈텔레메트리 네이티브 플랫폼
  - 설정과 통합
    - 오픈텔레메트리 데이터를 직접 수집하며 파이썬 자동 계측을 지원
  - 핵심 기능
    - 에이전트 트레이스에 대한 AI 기반 인사이트, LLM 지표용 커스텀 대시보드
  - 트레이드 오프
    - 장점
      - 단일 앱으로 단순한 설정, 소규모 팀에 낮은 오버헤드, 강력한 오픈텔레메트릭 지원, AI 확장
    - 단점
      - 생태계 확장성(플러그인 수) 한계
      - 시각화가 실용적이지만 고도화되지 못함
##### 10.2.5 랭퓨즈
- 파운데이션 모델과 에이전트 가시성에 특화, 에이전트용 의미 중심 트레이싱을 기존 스택에 손쉽게 확장
  - 설정과 통합
    - SDK로 랭그래프에 통합
    - 프롬프트, 출력, 평가를 캡처
  - 핵심 기능
    - LLM 네이티브 지표
    - 디버깅을 위한 세션 리플레이
  - 트레이드오프
    - 장점
      - 개발팀이 사용하기 쉬움
    - 단점
      - 범위가 좁아 프로메테우스와의 병행이 필요
      - 비 LLM 텔레메트리 확장성이 낮음
### 10.3 프로젝트에 적합 스택
- 신규 프로젝트라면 그라파나나 시그노즈 가 폭넓게 커버

| 스택           | 핵심 강점               | 적합한 대상    | 트레이드 오프(그라파나 대비) |
|--------------|---------------------|-----------|------------------|
| 그라파나 + 로키/템포 | 조합 가능성과 시각화         | 엔터프라이즈 운영 | 관리해야 할 컴포넌트가 많음  |
| ELK 스택       | 고급 검색/분석            | 대규모 로그    | 리소스 사용량 증가       |
| 피닉스          | 트레이싱과 디버깅           | 개발 반복     | 프로덕션 확장 한계       |
| 시그노즈         | 통합형이면서 경량           | 스타트업/ML팀  | 확장성 낮음           |
| 랭퓨즈          | 파운데이션 모델/에이전트 특화 평가 | 시멘틱 모니터링  | 인프라 커버리지 범위가 좁음  |

### 10.4 오픈텔레메트리 계측
- 효과적인 모니터링 루프를 구축하는 첫 단계는 계측
  - 오픈텔레메트리는 트레이스, 지표, 로그 전반에 걸친 구조화가 되고, 랭그래프 기반 에이전트 시스템과도 잘 통함
    - 랭그래프에서 스팬을 통해 구조화된 트레이스를 만듬
- 각 노드마다 함수 시작 시, 스팬 시작과 관련 메타데이터를 주석을 다는 것을 권장
- 계측 범위는 신중해야함
  - 정보가 지나치면 노이즈, 부족하면 근본 원인 분석이 어려움
- 템포는 트레이스 백엔드 역할
- 로키는 로그 집계 레이어
- 그라파나는 이 두데이터 스트림을 단일 화면으로 통합
### 10.5 시각화와 알림
- 로그와 트레이스를 로키와 템포로 스트리밍하면 그라파나로 시각화와 알림 처리
  - 에이전트의 의미론과 성공 기준에 맞춘 맞춤형 대시보드 구성
  - 커스텀 알림 구성
    -  최근 30분 동안 할루시네이션 비율이 5% 초과
    - 단일 세션에서 재시도 루프가 3회를 초과
    - 핵심 도구의 평균 응답 시간이 50% 초과 증가
### 10.6 모니터링 패턴
- 관측 가능성을 구축했으면 시스템을 어떻게 안전하게 변경 후 배포할까?
  - 실험의 리스크를 줄이고 프로덕션 변경에 안전망을 두는 '모니터링을 고려한 개발 패턴' 채택
##### 10.6.1 섀도 모드
- 새롭거나 실험적인 에이전트 버전을 현재 프로덕션 에이전트와 나란히 실행
  - 동일한 입력을 처리하지만 출력은 사용자에게 미제공 -> 작동 과정을 로그와 트레이스로 관찰
##### 10.6.2 카나리 배포
- 새 에이전트 버전을 실제 사용자 일부에게만 제공
  - 그라파나 대시보드를 통해 기존 에이전트와 비교
  - 작동이 양호하면 점진적으로 확대 -> 영향력 최소화
##### 10.6.3 회귀 트레이스 수집
- 에이전트는 프로덕션에서 실패할 때마다 학습 기회가 생김
  - 테스트 스위트로 자동 내보내면 지속적으로 갱신되는 회귀 코퍼스 구축
##### 10.6.4 자가 치유 에이전트
- 모니터링을 실패를 탐지하는 데 그치지 않고 에이전트 회복하도록 도움
  - 문제가 감지되면 폴백 메커니즘 실행
    - 도구 호출이 반복 실패하면 폴백 계획으로 우회하거나 사용자가에게 추가 설명을 요청
### 10.7 사용자 피드백
- 피드백은 바꿔 말하거나 작업 포기하는 암시적이거나, 엄지내림과 같은 명시적일 수도 있음 -> 두형태 모두 모니터랭해야함
  - 작업 포기율, 재질의 빈도 같은 암시적 피드백 지표를 시각화
  - 낮은 평점과 같은 명시적 피드백에서 불만 급증 시, 알림 트리거 구성
### 10.8 분포 변화
- 모니터링할 때 분포 변화를 식별하고 관리하는 건 어렵지만 결정적으로 중요한 과제
  - 시간이 지나며 에이전트 환경의 통계적 특성이 바뀔때 마다 발생
    - 명시적 에러보다 성능 저하, 불일치한 출력, 폴백 사용 증가로 나타남
- 모니터링 시스템은 느린 드리프트에 맞서는 1차 방어선
  - 할루시네이션 빈도 같은 시멘틱 지표를 추적하는 대시보드로 초기 신호 확인
- KS 검정
  - 통계 검정을 사용 해 입력 특성이나 출력의 분포를 비교
  - 두 데이터셋의 경험적 누적 분포함수를 비교해 동일한 분포에서 나왔는지 판단
  - 지연시간, 수치 지표처럼 연속형 특성의 변화를 탐지하는데 적합
- 쿨백-라이브러(KL) 발산
  - 확률 분포가 다른 분포와 얼마나 다른지를 측정하여 토큰 변화를 정량화해 드리프트를 탐지
- PSI
  - 범주형 또는 구간화된 연속형 변수에서 과거와 현재의 비율 분포를 비교해 변화를 탐지
  - 도구 호출, 정확도의 급감, 작업 포기율 증가 같은 에이전트 지표에 적합
- 복원력 있는 시스템을 구축하려면 변화에도 대응해야함
  - 일시적 변화는 입계값 조정이나 파싱 로직 업데이트로 흡수
  - 지속적 변화는 워크플로 재학습이나 새 API에의 적응이 필요

### 10.9 지표 소유권과 기능 간 거버넌스
- 에이전트 시스템을 배포하면 누가 어떤 지표를 소유하는가에 대한 이슈가 생김
  - 기존 소프트웨어 스택과는 다르게 파운데이션 모델이 구동하는 에이전트는 이런 경계가 없어야 함 -> `모니터링 전략도 그래야 함`
- 로그, 트레이스, 평가 시그널은 서비스 상태와 시스템 지표와 함께 관측되어야함
  - `파운데이션 모델의 응답은 단순한 모델 산출물이 아닌 제품`
  - `길게 이어지는 도구 호출, 재시도, 폴백은 사용자 경험`
- 각 팀은 에이전트 텔레메트리의 일부를 소유해야하며, 어느 팀도 이를 독립적으로 해석할 수 없음
- RACI
  - 지표 소유권의 조직적 과제를 다루기 위해 기능 전반의 역할을 명확하기 위해 사용
  - R(Responsible): 수행
  - A(Accountable): 결과 책임
  - C(Consulted): 자문
  - I(Informed): 통보

| 지표/활동              | 프로덕트 팀                    | ML 엔지니어                 | 인프라/SRE팀                  |
|--------------------|---------------------------|-------------------------|---------------------------|
| 지연시간               | A(사용자 영향 소유)/C(UX 임계값 자문) | R(프롬프트/모델 최적화)/I(회귀 통보) | R(인프라 원인 모니터링)/C(스케일링 자문) |
| 할루시네이션 비율          | C(사용자 피드백 컨텍스트 제공)/I(추세 통보) | A/R(평가를 통한 탐지·완화 소유)     | I(알림 설정을 위한 통보)            |
| 작업 성공률             | A(제품 목표 소유)/R(성공 기준 정의)   | C(모델 개선 자문)             | I(시스템 신뢰성과의 연계 통보)         |
| 토큰 사용량/비용          | C(비즈니스 영향 자문)             | R(생성 최적화)/I(급증 통보)      | A(예산/스케일링 소유)/R(인프라 효율 모니터링) |
| 분포 변화              | I(프로덕트 조정 통보)             | A/R(임베딩/평가로 탐지)         | C(데이터 파이프라인 안정성 자문)        |
